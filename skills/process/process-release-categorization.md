# Категоризация релизов: Task vs Skill vs Task&Skill

> **Context**: При обработке релизов из внешних источников (NEWS_SOURCES) каждый релиз должен получить корректную категорию. Неверная категоризация приводит к неправильной приоритизации работ.

## 1. Определения

### Task (Задача)
Релиз требует **конкретных действий** в коде, конфигурации или инфраструктуре проекта.

**Маркеры:**
- Новые фичи, которые нужно внедрить/настроить в НАШЕМ проекте
- Breaking changes, требующие миграции
- Security-патчи, требующие немедленного обновления зависимости
- Новая конфигурация, которую нужно добавить/изменить

### Skill (Знание)
Релиз содержит **информацию для принятия будущих решений**, но не требует немедленных действий.

**Маркеры:**
- Баг-фиксы в инструменте, который мы используем, но не затрагивают наш код
- Новые API/фичи в версии, на которую мы ЕЩЁ НЕ обновились
- Внутренние оптимизации без видимого эффекта
- Информация о known issues / regressions для мониторинга
- Обновления документации / best practices

### Task & Skill (Смешанная)
Релиз содержит **и** actionable items, **и** знания для будущего.

**Маркеры:**
- Часть изменений требует действий (security patch), часть — информационная
- Новые фичи, которые можно внедрить сейчас + знания о будущих возможностях

## 2. Алгоритм определения

```
1. ОПРЕДЕЛИ: используем ли мы эту ТОЧНУЮ ВЕРСИЮ?
   - ДА → переходи к п.2
   - НЕТ (мы на старой мажорной версии) → скорее всего Skill
   
2. ОПРЕДЕЛИ: есть ли BREAKING CHANGES или SECURITY PATCHES?
   - ДА → Task (или Task & Skill)
   
3. ОПРЕДЕЛИ: требуют ли новые фичи ИЗМЕНЕНИЙ в нашем коде/конфиге?
   - ДА → Task
   - НЕТ (просто стало лучше работать) → Skill
   
4. ОПРЕДЕЛИ: есть ли среди изменений МИКСт действий и знаний?
   - ДА → Task & Skill
```

## 3. Примеры правильной категоризации

| Релиз | Наша версия | Категория | Почему |
|-------|-------------|-----------|--------|
| Node.js 25.6.0 (новые API) | Node 20 | **Skill** | Мы на Node 20, новые API недоступны |
| Docker 29.2.1 (баг-фиксы) | Docker latest | **Skill** | Баг-фиксы применяются автоматически при обновлении |
| Continue AI 1.2.15 (subagents) | Continue latest | **Task** | Нужно настроить subagents и skills в конфиге |
| n8n 2.6.4 (security + fixes) | n8n latest | **Task & Skill** | OAuth security — проверить (Task), webhook fixes — знать (Skill) |
| Ollama 0.15.6 (баг-фиксы) | Ollama latest | **Skill** | Исправления работают автоматически |
| Zod 4.3.6 (minor fixes) | Zod 3.x | **Skill** | Мы на v3, v4 — мажорная версия, пока наблюдаем |

## 4. Scoring Guidelines (расширение)

Скор полезности ЗАВИСИТ от категории:

- **Task**: 70-100 (чем критичнее действие, тем выше)
- **Skill**: 40-70 (чем релевантнее знание, тем выше)
- **Task & Skill**: 60-90 (среднее между action и knowledge)

## 5. Промпт для LLM-скоринга

При вызове Mistral для скоринга, включать контекст проекта:

```
КОНТЕКСТ ПРОЕКТА:
- Node.js: v20 (Docker image node:20-slim)
- Zod: v3.24.1
- Docker: latest (host)
- n8n: latest (Docker)
- Continue AI: latest (Docker)
- Ollama: latest (host)

ПРАВИЛА КАТЕГОРИЗАЦИИ:
- "Task" = требует конкретных действий в НАШЕМ проекте
- "Skill" = информация для мониторинга, не требует немедленных действий
- "Task&Skill" = смесь: часть требует действий, часть — знания
- Если мы НЕ на той мажорной версии — это Skill, не Task
```

## 6. Hard Constraints

- НИКОГДА не назначать Task для фич в версии, которую мы не используем.
- ВСЕГДА проверять текущую версию зависимости перед категоризацией.
- Security patches ВСЕГДА как минимум Task (или Task & Skill).
- Чистые баг-фиксы без наших действий = Skill.
